
# レビュープロトコル

## 目的

-   レビュワーとレビュイーの間での齟齬をなくすこと
-   レビューに関する最低限の基準を設けること
    -   間違いとは何か、と悩んだ時に｢正しい形はこのドキュメントにあるこの部分だよね｣、と根拠を示せるようにしたい
-   開発時点での質を上げることによりレビューのコストを減らし、より良いレビューが出来るようにすること
-   更改時のコストを下げること

## 前提

-   レビュー時は以下を終了した状態にしておく
    -   テストを終えていること
        -   APIであればcurl叩いて正しいかのチェック(達成感を主張するためにコードを残しておく)
        -   フロント側であればスクショを取ってgithubに置く(レビュー時の視覚補助と達成感を主張するために)
    -   ビルドエラーがないこと
        -   SyntaxError等がないこと
        -   Eslint違反がないこと
            -   開発環境ではEslintのデフォルトルールを前提する。
    -   その他警告がないこと


## フォーマットと構造
-   preitterに準拠

## コメント
-   関数・クラスにはJSDocを付ける
    -   JSDocのテンプレートは以下のフォーマットを利用する
        ```ts
        /*
        * 説明書き
        *
        * @params 引数
        *
        * @return 戻り値
        *
        */
        ```

## 命名規則

-   明瞭な名前を付けること
    -   名前を見て、変数関数クラスの目的・機能を正確に表す名前にする。
        -   原則英語とし、動詞・形容詞・名詞の役割に即した命名とする。
        -   関数名には動詞..など
